# School Space — Client Data Migration & Seeding Guide

This document describes how to onboard a new client school into a fresh School Space deployment. It covers the end-to-end process of collecting data from the client, preparing a seed script, loading it into the database, and verifying the result before handing over access.

The demo seed lives at `backend/prisma/seed.js` and is the reference implementation. For a new client, clone that script, replace the demo data with the client's data, and follow the checklist below.

---

## 1. Overview

### When to use this guide

- New school purchases a School Space subscription and needs their baseline data loaded (school, classes, subjects, teachers, students, parents).
- Existing school migrating from another system (Google Sheets, legacy ERP, paper) to School Space.
- Staging → production data promotion for a tenant.

### What gets seeded

In FK-safe order:

1. `school` + admin `user` (admin user created first, then school references it via `adminId`).
2. `subject` — NEB curriculum subjects for the school.
3. `class` (Prisma model `Renamedclass`, DB table `class`) — grade + section pairs.
4. `teacher` + `user` (role `TEACHER`) → then `teachersubject` (M:N) and `class.classHeadId`.
5. `student` + `user` (role `STUDENT`) → then `enrollment` for the active session year.
6. `parent` + `user` (role `PARENT`) → then M:N `parentstudents` link.
7. School config flags (`ratingsEnabled`, `activeExamTerminal`, `activePerformanceSession`, etc.).

Transactional / operational tables (`attendance`, `exammark`, `assignment`, `submission`, `rating`, `potentialmetric`, `notification`, `fee`, `transaction`, `quizresponse`, `studymaterial`) are **not** seeded at onboarding. They accumulate through normal app usage. Historical exam marks can be backfilled via the seed script if the client provides them.

### What NOT to seed for a real client

- Demo study materials, quizzes, assignments.
- `potentialmetric` rows — these are auto-calculated when the admin runs `/run-calculation` at terminal end.
- `sessioncompletion`, `schoolexampublish` — generated by session/exam lifecycle actions.
- Demo passwords (`Demo@1234`). Use strong per-user generated passwords or force first-login password reset.

## 2. Prerequisites

- MySQL 8.0+ running and reachable.
- Dedicated database per client (multi-tenant isolation is per-school-row, but for a single-client install use one DB).
- `backend/.env` configured with `DATABASE_URL` and a 32+ char `JWT_SECRET`. See `SETUP.md`.
- Schema synced: `cd backend && npx prisma generate && npx prisma db push`.
- Node 18+, `bcrypt` available (already in `backend/package.json`).
- SMTP creds configured if you want verification / reset emails to work out of the box (not required for seeding).

## 4. Building the Client Seed Script

### 4.1 Clone the demo seed

```bash
cd backend/prisma
cp seed.js seed_client_<schoolcode>.js
```

### 4.2 Strip demo data, add CSV loaders

Replace the hardcoded `teacherDefs`, `studentDefs`, `parentDefs`, etc. arrays in `seed.js` with CSV reads. Example pattern using the built-in `fs` + a tiny CSV parser (no new dependency needed):

```js
const fs = require('fs');
const path = require('path');

function readCsv(filename) {
  const raw = fs.readFileSync(path.join(__dirname, 'client_data', filename), 'utf8');
  const [header, ...rows] = raw.trim().split('\n').map(l => l.split(','));
  return rows.map(r => Object.fromEntries(header.map((h, i) => [h.trim(), (r[i] || '').trim()])));
}
```

Put client CSVs under `backend/prisma/client_data/<schoolcode>/`. That folder is gitignored — do **not** commit client PII.

### 4.3 Keep the `clean()` function — but gate it

The demo `clean()` deletes rows by hardcoded demo usernames. For client data, **delete this function entirely** or replace its body with a hard abort:

```js
async function clean() {
  throw new Error('Client seed is append-only. Use prisma db push --force-reset to wipe if needed.');
}
```

Destructive wipes on a client DB must be an explicit, separate, manually-invoked step — never part of the seed script.

### 4.4 Creation order in `main()`

Preserve this exact order. Later steps reference earlier IDs.

1. Admin `user` → `school` (with `adminId`) → update admin user's `schoolId`.
2. Loop subjects → `prisma.subject.create`.
3. Loop classes → `prisma.renamedclass.create`.
4. Loop teachers:
   a. Create `user` (role `TEACHER`, `schoolId` set, `emailVerified: true`, `isActive: true`).
   b. Create `teacher` (`status: 'ACTIVE'`, `userId`, `schoolId`).
   c. For each `subject × class` pair → `teachersubject.create`.
   d. If `classHead` set → `renamedclass.update` with `classHeadId`.
5. Loop students:
   a. Create `user` (role `STUDENT`).
   b. Create `student` with `studentCode`, `rollNo`, `classId`, `isApproved: true`.
   c. Create `enrollment` row for `activePerformanceYear`.
6. Loop parents:
   a. Create `user` (role `PARENT`).
   b. Create `parent` with `student: { connect: [...] }` to link children.
7. Optional: loop exam marks CSV → `exammark.create` with the correct `examTerminal`.

### 4.5 Password handling

- **Do not** reuse the demo `Demo@1234` password.
- Generate a random 12-char password per user (`crypto.randomBytes(9).toString('base64')`), write `{ username, tempPassword }` pairs to `backend/prisma/client_data/<schoolcode>/credentials.csv`, hash before storing in DB.
- Hand the credentials CSV to the school admin over a secure channel (1Password share, encrypted email). Delete the file after handover.
- `user.emailVerified` can be set to `true` during seed to skip the email-verification flow, or `false` to force users through it on first login.

### 4.6 Dry-run

Before running against the real DB, point `DATABASE_URL` at a scratch DB and run:

```bash
node prisma/seed_client_<schoolcode>.js
npx prisma studio   # visually inspect in browser
```

## 5. Executing the Migration

### 5.1 Pre-flight

- Confirm `backend/.env` `DATABASE_URL` points at the client's production DB.
- `npx prisma db push` — schema is current.
- `SELECT COUNT(*) FROM user;` — database is empty (or contains only your platform-admin row).
- Back up the DB: `mysqldump -u root -p schooldb_<client> > backup_preseed_$(date +%Y%m%d).sql`.

### 5.2 Run

```bash
cd backend
node prisma/seed_client_<schoolcode>.js 2>&1 | tee seed_<schoolcode>_$(date +%Y%m%d).log
```

The log file is your audit trail — keep it alongside the backup.

### 5.3 Verification queries

Run these against the client DB and paste results in the handover doc:

```sql
SELECT id, name, code, adminId, activeExamTerminal, activePerformanceSession, activePerformanceYear FROM school;
SELECT role, COUNT(*) FROM user GROUP BY role;
SELECT COUNT(*) AS classes FROM class;
SELECT COUNT(*) AS subjects FROM subject;
SELECT COUNT(*) AS teachers FROM teacher WHERE status = 'ACTIVE';
SELECT COUNT(*) AS students FROM student WHERE isApproved = 1;
SELECT COUNT(*) AS parents FROM parent;
SELECT c.name, c.section, COUNT(s.id) AS student_count
  FROM class c LEFT JOIN student s ON s.classId = c.id
  GROUP BY c.id;
SELECT c.name, c.section, t.id AS classHeadId FROM class c LEFT JOIN teacher t ON c.classHeadId = t.id;
```

Counts should match the client CSV row counts exactly. Any class missing a `classHeadId` will block attendance and exam-mark entry — fix before go-live.

### 5.4 Smoke test in the app

- Log in as the admin with the handover credentials → verify dashboard shows correct class/student counts.
- Log in as one teacher → verify assigned classes + subjects show up.
- Log in as one student → verify class + roll number correct.
- Log in as one parent → verify child(ren) visible.
- Admin → Sessions → confirm `activePerformanceSession` matches what the client specified.

## 6. Go-Live Checklist

- [ ] School record created with correct `code` (the code teachers/students use at signup).
- [ ] Admin user logs in successfully.
- [ ] All class heads assigned.
- [ ] `activeExamTerminal`, `activeRatingSession`, `activePerformanceSession` set to the correct current values.
- [ ] Feature flags (`ratingsEnabled`, `studentAnalyticsEnabled`, `parentMessagingEnabled`, `multiClassTeachersEnabled`) match the client's subscription tier.
- [ ] Credentials CSV delivered to the school admin via secure channel and deleted from local disk.
- [ ] Seed log + DB backup archived.
- [ ] Optional historical exam marks loaded, and admin has run `/run-calculation` for the imported terminal.
- [ ] `backend/prisma/client_data/<schoolcode>/` is gitignored.

## 7. Rollback Plan

If seeding produced bad data and the DB must be wiped:

```bash
# 1. Confirm you have the pre-seed backup
ls -lh backup_preseed_*.sql

# 2. Wipe + restore schema
mysql -u root -p -e "DROP DATABASE schooldb_<client>; CREATE DATABASE schooldb_<client>;"
cd backend && npx prisma db push

# 3. Fix the seed script, dry-run on scratch DB, then re-run against client DB
node prisma/seed_client_<schoolcode>.js
```

**Never** run `clean()` (or the `mysqldump | grep -v … | mysql` tricks) on a DB that has any real usage on it. Once the client starts logging in, attendance / marks / messages accumulate and cannot be re-derived from CSVs.

## 8. Post-Onboarding: Incremental Data Changes

After go-live, use the admin UI — not the seed script — for additions:

| Change | Where in admin UI | Backend route |
|---|---|---|
| Add class | Admin → Classes | `POST /admin/classes` |
| Add subject | Admin → Subjects | `POST /admin/subjects` |
| Assign teacher to class+subject | Admin → Classes → Assign Teacher | `POST /admin/classes/assign-teacher` |
| Assign class head | Admin → Classes → Assign Class Teacher | `POST /admin/classes/:id/assign-class-teacher` |
| Add teacher / student / parent | They self-sign-up with school code `SHRE`, admin approves | `POST /auth/signup/{teacher,student,parent}` |
| Remove user | Admin dashboard → delete | `DELETE /admin/{teachers,students}/:id` |
| Start / end session | Admin → Sessions | `POST /admin/start-session`, `POST /admin/end-session` |
| Promote / retain / graduate | Admin → Promotions | `POST /admin/promotions/:studentId/{promote,retain,graduate}` |

See `ADMIN_FLOW.md` for the full admin lifecycle, including the Class 10 early-graduation rule and session-end promotion workflow.
